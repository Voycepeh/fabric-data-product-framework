# Handover Guide

Handover quality should be consistent across teams and independent of who built the pipeline.

## Required handover content

- Business purpose
- Source system
- Target table
- Refresh schedule
- Data owner
- Technical owner
- Table dictionary
- Column dictionary
- DQ checks
- Schema drift behavior
- Data drift behavior
- Pipeline failure behavior
- Known limitations
- Support and escalation path

## Handover quality principle

The handover pack should be generated from metadata wherever possible, so handover quality does not depend on one engineer manually writing everything from memory.

## Suggested handover outputs

- Runbook summary
- Test plan summary and latest validation status
- Contract references (data + pipeline)
- Operational support notes and escalation contact path
